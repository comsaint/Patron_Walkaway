Patron Walkaway Predictor：專案簡報（方法與理由）

> 這份文件的目標是讓團隊快速理解：我們要解什麼問題、我們用什麼方法解、以及為什麼這是一個能落地、能迭代的選擇。  
> 深入驗證（SQL、完整規格、過去踩雷）請看文末「深入細節」與附錄。

---

## 1. 我們要解的問題

Smart Table 讓投注行為能夠近乎即時被記錄。我們希望把這件事轉成一個可用的營運能力：**在玩家真正離開之前，提前提醒公關介入挽留**。

- **輸入**：玩家在桌台上的投注行為
- **輸出**：一個可操作的「離場預警」
- **限制**：預警必須能在線上以當下可得資料重現（不能靠事後才知道的資訊）

---

## 2. 我們怎麼定義「離場」與「可行動的預警」

為了可量化、可驗證、可上線，我們用行為定義離場事件：

- **離場 (Walkaway)**：從某時間點起，玩家至少 **30 分鐘**沒有再下任何注
- **預警視窗**：我們希望在離場前的 **15 分鐘內**發出提醒

因此在每一次下注時點，我們都在回答同一個問題：

> 這位玩家是否很可能在接下來 15 分鐘內停止下注，並且至少 30 分鐘不會回來？

---

## 3. 我們的整體做法

我們刻意走「先能落地、再變強」的路線，確保訓練與線上一致：

- **用投注事件當主時間軸**：因為它最接近即時、可穩定取得
- **先把資料整理到可信**：去重、排除非行為資料、身份歸戶
- **用近期行為訊號建特徵**：例如節奏變慢、金額變化、輸贏壓力等
- **用可解釋且成熟的表格模型做 MVP**：快速上線、便於跟業務溝通
- **用偏保守的警報策略建立信任**：降低假警報對採用的傷害

**為什麼特徵工程先以「近期行為」為主**：近期窗口的節奏/金額/輸贏等特徵最能反映即時狀態，也與線上推論時的資料可得性一致。

---

## 4. 我們實際對資料做了哪些處理

### 4.1 Session 資料（用於身份歸戶與歷史輪廓）

- **去重**：同一筆 session 只保留最新版本，避免重複計算
- **排除人工帳務調整**：避免非打牌行為污染行為分佈
- **排除刪除/取消紀錄**：避免撤銷資料被當成真實行為
- **清洗會員卡號缺失**：把 `'null'` 等視為空值，避免歸戶錯誤
- **以「session 結束後可得」的語義使用 session**：session 在結束後才入湖，推論時需依此語義

另外，基於 session 我們也**實際做了**：
- **身份歸戶（Canonical ID）**：把同一位玩家串起來（原因與證據見附錄 A.1）
- **排除明顯 Dummy IDs**：避免假帳號稀釋行為訊號

### 4.2 Bet 資料（我們的即時主訊號）

- **以派彩完成時間作為事件時間（並保留合理延遲）**
- **穩定排序**：確保同一時間點多筆下注在訓練/線上計算結果一致

### 4.3 Player 日快照（歷史輪廓；離線產生）

- **先完成 session 的清洗後再彙總**：避免把污染帶入長期輪廓
- **只用 as-of（PIT-safe）方式取用**：各下注時點只取該時點之前的最新快照（PIT = Point-in-Time）

### 4.4 管線層（標籤正確性）

- **延伸拉取（只為了判斷窗口末端是否真的離場）**：避免把「資料切窗結束」誤當成離場

---

## 5. 特徵工程

### 5.1 我們做了什麼

**雙軌架構**：軌道 A（Featuretools DFS）＋ 軌道 B（手寫向量化），兩軌共用 `cutoff_time`，產出後合併。

| 軌道 | 性質 | 產出 | 說明 |
|------|------|------|------|
| **軌道 A** | Automatic Feature Generation (Featuretools DFS) | 聚合、窗口、組合特徵 | 從「下注、桌台時段、玩家歷史」等表自動算出「過去 5／15／30 分鐘內的下注次數、金額、趨勢」等數字。有卡客可串出歷史；無卡客只看當次下注。因資料量大，先抽樣試算、把規則存起來，再用同一規則對全部資料算一遍，確保訓練與上線一致。 |
| **軌道 B** | Manual-craft features | `loss_streak`、`run_boundary`、`minutes_since_run_start` | FeatureTools 難以表達的狀態：例如「目前連續輸了幾把」（贏了就歸零、和局不算）、「這把距離這一輪連續下注開始已經過了幾分鐘」（中間休息超過 30 分鐘視為新的一輪）。訓練與上線用同一段程式碼計算。 |
| **Feature screening** | 軌道 A 候選的後處理 | 候選篩選 | 軌道 A 會產出很多特徵，先篩掉對預測沒幫助/重複的，只保留有價值的留給模型用。 |

**術語**：**Run** = 同一玩家連續下注段，相鄰 bet 間距 ≥ 30 分鐘即切分為新 run。

### 5.2 我們捕捉的訊號類別

- **下注節奏**：近期頻率、間隔變化
- **金額與風險**：注額變化、縮手／加大
- **輸贏壓力**：連輸、近期淨輸贏
- **有卡客專用**：跨日／跨桌歷史輪廓（PIT-safe 快照）

上述訊號均在觀測時點可算，訓練與線上一致。

### 5.3 為何這樣設計

- **雙軌**：Featuretools 擅長系統化探索；狀態機（`loss_streak`、`run_boundary`）用 Custom Primitive 成本高且易錯，改手寫。
- **兩階段 DFS**：全量 4.38 億筆無法一次 DFS；抽樣探索 → 持久化定義 → 全量計算，同時避免 train-serve parity 風險。
- **不用 session 結束特徵**：session 結束在離桌後才入湖，線上不可得。

## 6. 建模策略

### 6.1 我們做了什麼

| 面向 | 做法 |
|------|------|
| **模型** | 兩個模型：Rated（有卡客）、Non-rated（無卡客）；各自 LightGBM，各自閾值。 |
| **演算法** | LightGBM，`class_weight='balanced'`。 |
| **超參調優** | Optuna（TPE Sampler）搜尋 `n_estimators`、`learning_rate`、`max_depth` 等，目標 average precision (AP)。 |
| **閾值選擇** | validation 上掃閾值取 F1 最大；Backtester 雙模型閾值同以 Optuna 2D 搜尋 F1，並受 G1 約束（precision 下限、最小警報量）。 |
| **資料切分** | 嚴格按時間：Train 60–70%、Valid 15–20%、Test 15–20%，不可隨機。 |
| **評估指標** | Bet-level：Precision、Recall、AP (average precision)、F1。 |

### 6.2 為何這樣設計

- **雙模型**：有卡客有 session／player 歷史；無卡客僅 bet。併成一模型會大量 NaN、偏差大；分開可各自校準閾值。
- **LightGBM + Optuna**：速度快、可解釋（SHAP），Phase 1 目標是儘快落地與可回顧，不引 AutoML。
- **不先做序列深度模型**：最大風險在資料與時間語義（線上不可重現），不在模型容量。先用穩定基線打通 pipeline，Phase 2+ 再評估序列嵌入。

---

## 7. 我們怎麼讓警報可用（Alert strategy）

初期採保守策略：
- **寧可少一點警報，也要先確保警報可信**
- 之後依公關承載量與業務偏好，逐步調整到「既可用、也可負擔」的警報量

**為什麼這樣做**：假警報會快速消耗現場信任，導致系統即使「理論上有用」也不會被使用。因此我們先把早期版本設計成偏保守、可回顧、可校準的預警機制，再用實際營運回饋逐步調整警報量與門檻。

本節用白話說明專案中若干「我們如何從資料推論出結論」的調查，供非技術背景者理解。

### A.1 player_id 與 casino_player_id 的映射關係

**我們想回答的問題**：  
`player_id`（桌台辨識系統給的「這個人」ID）和 `casino_player_id`（會員卡號）之間，是一對一、還是一對多、還是多對多？

**白話解釋兩個 ID 的來源**：
- **player_id**：Smart Table 的生物辨識或桌台追蹤系統，根據「誰坐在這個位置打牌」產出的識別碼。系統重啟、鏡頭調整、或辨識斷線時，同一人可能被重新指派一個新的 ID。
- **casino_player_id**：玩家刷會員卡時帶出的卡號，理論上是同一張卡對應同一個人。

直覺上，我們可能假設：有刷卡的玩家，一張卡對應一個「桌台 ID」，應該是 1:1。若真是如此，用任一個 ID 做歸戶結果會一樣。

**我們怎麼驗證**：
1. 先清洗資料（把字串 `'null'`、空字串當成真正的「沒有」）。
2. 對每個 `player_id`，統計它曾經關聯過**幾張不同的卡號**（同一人換卡會造成 1 player_id → 多張卡）。
3. 對每個 `casino_player_id`，統計它曾經關聯過**幾個不同的 player_id**（系統重新指派 ID 會造成 1 張卡 → 多個 player_id）。

可重現的驗證 SQL 見 `doc/FINDINGS.md` 附錄 **[FND-11]**。

**我們得到的結論**：
- 有 **97 個 player_id** 對應到多張卡：代表同一人（同一桌台辨識 ID）在不同時間刷了不同卡，例如補卡、換卡。
- 有 **97 張卡** 對應到多個 player_id：代表同一張卡在不同時間被系統辨識成不同的「桌台 ID」，例如生物辨識斷鏈後重新建檔。

雖然數量只佔約 0.03%，但一旦發生，會造成：
- 若只用 `player_id` 歸戶：同一個人換卡後，其歷史會被切成多段，看起來像「多次短暫停留後離開」，產生大量假 Walkaway 標籤。
- 若不做處理直接 JOIN：會產生笛卡兒積式的重複計算，玩家輪廓被撕裂。

**因此我們採取的決策（D2 Canonical ID）**：  
有卡客以 `casino_player_id` 為唯一錨點，將所有曾關聯過的 `player_id` 紀錄彙整上去；無卡客才退而求其次用 `player_id`。

**M:N 衝突的具體處理規則**（詳見 `ssot/patron_walkaway_phase_1.plan.md` Step 2）：
- **情境 1（斷鏈重發）**：同一張卡對應多個 player_id → 全部映射至同一個 canonical_id（= 該卡號）。
- **情境 2（換卡）**：同一個 player_id 對應多張卡 → 取最近一次（最大 `lud_dtm`）的卡號為 canonical_id；衝突清單寫入稽核日誌。
